## Git運用ルール

### ブランチ運用

#### main

- 本番用ブランチ
- Netlifyなどのデプロイ対象
- 直接コミットは禁止（GitHub のブランチ保護設定などで制限）
- `develop` からのマージのみ

#### develop

- デフォルトブランチ
- `feature` ブランチをマージする先
- 機能がまとまったら `main` にマージ

#### feature/\*

- 機能ごとの作業ブランチ
- `develop` から作成し、作業完了後 `develop` にマージ
- ex: issue番号が45の場合、`feature/45`

### マージフロー

```
feature → develop → main
```

- `feature` → `develop`
  機能実装完了時にPRを作成

- `develop` → `main`
  リリース単位でPRを作成

### コミットメッセージ

issue番号が45の時

```
refs: #45 内容
```

### issueについて

- レビュワーが、そのissueで何をやったかを把握できる程度の情報を書く
- コードの内容と変更差分に問題がなければマージする

### プルリクエスト（PR）

#### PR作成時のルール

**タイトル**

- issue番号と簡潔な内容を記載

```
例: refs #45 ヘッダーコンポーネントの実装
```

**説明（Description）**

- 以下の内容を含める：
  1. **何を実装したか**
     - 実装内容の概要
  2. **変更内容**
     - 主な変更点をリスト形式で記載
  3. **確認方法**
     - レビュワーが動作確認する方法（該当する場合）
  4. **スクリーンショット**
     - UIの変更がある場合は必ず添付

**例:**

```markdown
## 概要

#45 ヘッダーコンポーネントを実装しました。

## 変更内容

- Header.jsxを新規作成
- レスポンシブ対応のハンバーガーメニューを実装
- ナビゲーションリンクを追加

## 確認方法

1. `npm run dev` でローカルサーバーを起動
2. ヘッダーが表示されることを確認
3. ハンバーガーメニューが動作することを確認
```

#### レビュー依頼

- PRを作成したら、適切なレビュワーをアサイン
- Discordでメンションして連絡

#### レビュー対応

- レビューコメントには必ず返信する
- 修正が必要な場合は修正後にコメントで報告
- 追加修正時には新規commit or amendで変更を追加

#### マージ

- レビュワーがマージ処理を行う

## プロジェクト構造・ディレクトリ規則

### 基本構造

```
officialWebApps/
├── public/          # 静的アセット（画像、favicon等）
├── src/
│   ├── components/  # 再利用可能なコンポーネント
│   ├── App.jsx      # ルートコンポーネント
│   ├── main.jsx     # エントリーポイント
│   └── index.css    # グローバルスタイル
├── documents/       # プロジェクトドキュメント
└── ...
```

### ディレクトリ規則

#### コンポーネントの配置

- **再利用可能なコンポーネント**: `src/components/` に配置
- **ページレベルのコンポーネント**: `src/` 直下に配置（例: `Hero.jsx`, `App.jsx`）

##### 複数ファイルを持つコンポーネント

1機能について2つ以上のファイルがある場合は、専用のディレクトリを作成してまとめる：

```
// Good: 複数ファイルがある場合
src/components/Header/
├── Header.jsx
├── Header.css
└── HeaderMenu.jsx  // サブコンポーネント

// Good: 単一ファイルの場合
src/components/
├── Hero.jsx
└── Contact.jsx
```

**ディレクトリ構造の例:**

```
src/components/Header/
├── Header.jsx       // メインコンポーネント
├── Header.css       // スタイル
└── HeaderMenu.jsx   // 関連するサブコンポーネント
```

#### 機能別ディレクトリ

必要に応じて以下のディレクトリを追加可能：

- `src/hooks/` - カスタムフックを配置
- `src/utils/` - 汎用的なユーティリティ関数を配置
- `src/constants/` - 定数を配置

#### 静的アセット

- **画像・アイコン**: `public/` に配置
- **favicon等**: `public/` に配置

## ファイル命名規則

### コンポーネントファイル

- **React コンポーネント**: PascalCase を使用
  - 例: `Header.jsx`, `MainContent.jsx`, `Hero.jsx`
  - 拡張子は `.jsx` を使用

### CSSファイル

- **スタイルファイル**: コンポーネント名に合わせる、または `kebab-case`
  - 例: `Splash.css`, `index.css`

### その他のファイル

- **画像ファイル**: 特に厳密な規則は設けない
  - 内容がわかりやすい名前であれば良い
- **設定ファイル**: 各ツールの慣例に従う

## コーディングスタイル

### JavaScript/React

#### 基本スタイル

- **インデント**: 2スペース
- **クォート**: ダブルクォート (`"`) を使用
- **セミコロン**: 文末に必須
- **関数宣言**:
  - コンポーネントや通常の関数宣言: `function` キーワードを使用
  - 小さいコールバック関数: アロー関数を使用可能
- **Export**: Named Exportを使用（default exportは使用しない）

  ```jsx
  // Good - コンポーネントはfunction宣言
  export function MyComponent() {
    return <div>Hello</div>;
  }

  // Good - 通常の関数もfunction宣言
  export function calculateTotal(items) {
    return items.reduce((sum, item) => sum + item.price, 0);
  }

  // Good - コールバックはアロー関数OK
  export function UserList({ users }) {
    return (
      <ul>
        {users.map((user) => (
          <li key={user.id}>{user.name}</li>
        ))}
      </ul>
    );
  }
  ```

#### import/export文の規則

##### export文

- **Named Export を使用**（default export は使用しない）
- コンポーネントや関数は定義と同時にexportする

```jsx
// Good - コンポーネント
export function MyComponent() {
  return <div>Hello</div>;
}

// Good - 通常の関数
export function myFunction() {
  // ...
}

// Good - イベントハンドラーなど、内部でアロー関数を使用
export function UserCard({ user }) {
  const handleClick = () => {
    console.log("Clicked:", user.name);
  };

  return <div onClick={handleClick}>{user.name}</div>;
}
```

#### コメント

- 複雑なロジックには説明コメントを追加
- コンポーネントの役割が不明瞭な場合は、冒頭にコメントを記載
- TODO コメントはissueを発行し、issue番号を記載

```jsx
// TODO: #issue番号 この部分をリファクタリング予定
```

## コンポーネント設計規則

### 基本原則

- **1ファイル1コンポーネント**: 基本的に1つのファイルには1つのコンポーネントのみ定義
- **コンポーネントの分割**: 再利用可能性や責務の分離を意識して適切に分割
- **コンポーネント名**: ファイル名と同じ名前を使用

### Props の命名規則

- **camelCase** を使用
- boolean型のpropsは `is`, `has`, `should` などの接頭辞を使用

```jsx
<Header isOpen={true} hasMenu={true} />
```

### State 管理

- **useState**: ローカルステートの管理に使用
- 複雑なステート管理が必要になった場合は、Context API等の導入を検討

### Hooks使用時の注意

- Hooksはコンポーネントのトップレベルで呼び出す
- 条件分岐やループ内でHooksを呼び出さない
- カスタムHooksは `use` で始まる名前にする

## CSS/スタイリング規則

### スタイリング手法

- **Plain CSS** を使用
- 必要に応じて将来的に CSS Modules 等への移行を検討可能

### クラス命名規則

- **kebab-case** を使用

```css
.site-header {
  /* ... */
}

.menu-toggle {
  /* ... */
}

.site-logo-img {
  /* ... */
}
```

## レビュー・マージ基準

### レビュー時のチェックポイント

#### コードレビューで確認すべき項目

1. **ガイドラインへの準拠**
   - ファイル命名規則、コーディングスタイルに従っているか

2. **コードの品質**
   - 適切にコンポーネントが分割されているか
   - 冗長なコードがないか
   - エラーハンドリングが適切か

3. **パフォーマンス**
   - 不要なレンダリングが発生していないか
   - 画像の最適化がされているか

4. **アクセシビリティ**
   - 適切な HTML タグが使用されているか
   - `alt` 属性や `aria-label` が適切に設定されているか

5. **コミットメッセージ**
   - issue番号が適切に記載されているか（`refs: #XX`）

### マージ基準

- コードの内容と変更差分に問題がないこと
- issueの内容が適切に実装されていること
- レビュワーが変更内容を理解できること
